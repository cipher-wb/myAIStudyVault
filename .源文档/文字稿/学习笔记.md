# **Git和Github的区别**

##  

## **Git**

![img](https://wdcdn.qpic.cn/MTMxMDI3MDE1NDg0OTIxMzA_232995_FMJLo206ak3nlBM0_1773220975?w=2866&h=1210)

 

​                ● **定义：**一个分布式版本控制系统

​                ● **功能：**追踪文件变化历史，管理代码版本

​                ● **工作方式：**在本地创建代码仓库，记录所有更改

​                ● **主要特点：**

​                ○ 完全离线工作，不需要网络连接

​                ○ 快速的分支创建和合并

​                ○ 强大的历史记录和版本回溯能力

​                ● **使用场景：**单人开发或团队协作的代码版本管理

 

## **Github**

![img](https://wdcdn.qpic.cn/MTMxMDI3MDE1NDg0OTIxMzA_645443_gqCfLKelFjQiW30W_1773220975?w=3694&h=1812)

​                ● **定义：**基于Git的云端托管服务

​                ● **功能：**远程存储Git仓库并提供协作工具

​                ● **工作方式：**提供云端存储空间和Web界面来管理Git仓库

​                ● **主要特点：**

​                ○ 远程代码仓库托管

​                ○ Pull Requests（拉取请求）实现代码审查

​                ○ Issues（问题）追踪系统

​                ○ Actions（自动化工作流）

​                ○ 社区和协作功能

​                ● **使用场景：**团队协作、开源项目、代码分享

 

# **Git常见命令**

 

Git 是一个版本控制系统，帮助你跟踪代码的变化。以下是日常使用的核心命令：

 



**Git最重要的几个命令：**

​                ● **git init**：初始化一个新的Git仓库

​                ● **git clone**：克隆现有的仓库

​                ● **git add** ： 将文件添加到暂存区

​                ● **git commit**：提交暂存区的更改

​                ● **git push** ：推送本地更改到远程仓库

​                ● **git status** ：查看工作区状态

​                ● **git pull** ：拉取远程更改并合并

​                ● **git branch** ：查看、创建或删除分支

​                ● **git checkout** ：切换

​                ● **git switch**：切换分支

​                ● **git merge** ：合并分支





 

你目前可能不知道这些命令是干什么，没关系。接下来我会带你去认识这些操作：

​            1.     你可以完全使用UI，不去了解这些命令的具体含义

​            2.     你也可以像一个极客一样使用这些命令。

## **4. 分支管理**



**为什么要有分支？** 

想象这么一个场景，如果你的项目上线了。你现在正在开发新的功能，开发到一半的时候，发现线上有一个非常紧急的bug需要处理。那这个时候你就要在当前开发到一半的分支的基础上去处理这个bug。

 

那问题来了，如果说你处理完这个bug，然后把这个项目推送上线的话，那你同时也把你这开发到一半的功能推送上线给用户了。

 

那应该怎么解决这个问题呢？这个就需要分支来解决！

#             **3.**     **基于git命令的教学实战**

 

## **创建和获取仓库**

 

### **git init****：开始一个新项目**

 



```
git init 
```





这会在当前文件夹创建一个新的 Git 仓库。

 

### **git clone****：获取已有项目**

 



```
git clone https://github.com/用户名/项目名.git 这会下载一个完整的项目副本到你的电脑。
```

 



 

## **日常工作流程**

 

### **git status****：查看当前状态**

 



```
 git status 
```





显示哪些文件被修改、暂存或未跟踪。

###  

### **git add****：添加暂存区**

 



```
git add 文件名    # 添加单个文件 
git add .        # 添加所有更改 
```

 



将文件放入"暂存区"，准备提交。

###  

### **git commit****：提交更改**

 



```
git commit -m "描述你做了什么更改" 
```

 



将暂存区的内容保存为一个新的提交记录。

###  

### **git log****：查看提交日志**



```
 git log 
```





查看你所有的提交的日志，会显示你git commit的信息。

##  

## **远程协作**

 

### **git push****：上传更改**

 



```
git push origin main 
```

 



将本地提交推送到远程仓库（例如 GitHub）。

 

### **git pull****：获取更新**



```
git pull 
```

 



从远程仓库下载更新并合并到本地。

 

## **分支管理**

 

### **git branch****：分支操作**

 

### `git branch           # 查看分支列表 git branch 新分支名   # 创建新分支 git branch -d 分支名  # 删除分支 ` 

###  

### **git checkout****：检出与切换**

 



```
git checkout 提交ID     # 查看历史版本 
git checkout 分支名      # 切换到某分支（官方不再推荐，但依然大规模使用） 
git checkout -b 新分支名 # 创建并切换到新分支（官方不再推荐，但依然大规模使用） 
```

 



###  

### **git switch****：现代分支切换（新版Git）**

 

### `git switch 分支名       # 切换分支 git switch -c 新分支名  # 创建并切换到新分支 ` 

###  

### **git merge****：合并分支**



```
git switch main       # 先切到目标分支 
git merge 特性分支名    # 将特性分支合并到当前分支
```





##   

##  

## **退回某个版本**

 

### **git reset****：退回到某个提交**



```
git reset 提交ID 
```

 



##  

## **实用小贴士**

 

​                ● 使用 git status 随时检查工作状态

​                ● 经常提交，提交信息要有意义

​                ● 使用分支进行新功能开发

​                ● 合并前先拉取最新代码

 

这些基本命令可以满足大部分日常开发需求，熟练掌握后会让你的开发工作更高效。

要理解Next.js，首先需要简单了解React：



**React** 是一个用于构建用户界面的JavaScript库。它的核心理念是将界面拆分为可重用的组件，每个组件管理自己的状态和渲染逻辑。React让开发者能够用声明式的方式描述界面，但它主要关注的是UI层面。





 



**Next.js** 则是建立在React基础上的框架，它扩展了React的能力：

​                ● React提供了组件系统和状态管理

​                ● Next.js添加了路由、渲染优化、构建工具等

​                ● Next.js简化了许多React应用中需要单独配置的事项





 

这种关系类似于：**React是引擎，而Next.js是一辆配备了引擎、车身、轮子和方向盘的完整汽车**。

### **.jsx是什么？**

在Next.js项目中，你会看到许多.jsx或.tsx文件。这些是React使用的特殊文件格式：

​                ● **JSX (JavaScript XML)** 允许在JavaScript中编写类似HTML的代码

​                ● **TSX** 是TypeScript版本的JSX，增加了类型检查功能



即使你不熟悉jsx的具体语法也没关系，使用Cursor等AI工具时，你只需描述你想要的功能，AI会为你生成正确格式的代码。

# **2. 项目结构与路由系统**

## **2****.****1** **Next.js的基本目录结构**

Next.js提供了两种组织项目的方式：App Router（app/目录）和Pages Router（pages/目录）。从Next.js 13开始，App Router是官方推荐的新方式，本章将主要介绍这种模式。

 

### **App Router 结构（官方推荐）**

 



```
my-nextjs-project/
├── app/                 # 应用主目录
│   ├── favicon.ico      # 网站图标
│   ├── globals.css      # 全局样式
│   ├── layout.tsx       # 根布局组件（应用于所有页面）
│   ├── page.tsx         # 首页组件
│   └── about/           # 关于页面目录
│       └── page.tsx     # 关于页面组件
├── components/          # 可复用组件目录（自创建）
├── public/              # 静态资源目录
└── ...                  # 其他配置文件
```





###  

### **核心文件说明**

​                ● **app/page.tsx**: 网站首页，对应路径 /

​                ● **app/layout.tsx**: 根布局，包含所有页面共享的结构

​                ● **app/globals.css**: 全局CSS样式

​                ● **app/[目录名]/page.tsx**: 子页面，对应路径 /[目录名]

###  

### **Pages Router 结构（不推荐）**

虽然不再是推荐方式，但你可能在许多现有项目中看到这种结构：

 



```
my-nextjs-project/
├── pages/               # 页面目录
│   ├── _app.js          # 全局App组件
│   ├── _document.js     # 自定义文档
│   ├── index.js         # 首页
│   └── about.js         # 关于页面
├── styles/              # 样式目录
├── public/              # 静态资源目录
└── ...                  # 其他配置文件
```





对于新项目，建议使用App Router（app/目录）结构，它提供了更好的性能和更丰富的功能。

## **2****.****2** **文件系统路由：自动URL映射**

Next.js的一个核心特点是**文件系统路由**。这意味着你的项目文件结构直接决定了网站的URL路径，不需要额外的路由配置。

### **基本路由规则**

在App Router中，路由遵循以下规则：

​            1.     **文件命名**：特殊文件有特定用途：

​                ● page.tsx: 定义路由的主要内容

​                ● layout.tsx: 定义共享UI

​                ● loading.tsx: 加载UI

​                ● error.tsx: 错误UI

​                ● not-found.tsx: 404 UI

​            2.     **目录映射**：目录名直接映射到URL路径：

 



```
app/page.tsx              → /
app/about/page.tsx        → /about
app/blog/page.tsx         → /blog
app/blog/post/page.tsx    → /blog/post
```





​            3.     **嵌套路由**：子目录创建嵌套路由，共享父布局：

 



```
app/layout.tsx            → 应用于所有页面
app/blog/layout.tsx       → 应用于所有/blog/*页面
```





###  

### **示例**

一个简单网站的文件结构：

 



```
app/
├── page.tsx         # 首页 (/)
├── layout.tsx       # 根布局
├── about/
│   └── page.tsx     # 关于页 (/about)
└── blog/
    ├── page.tsx     # 博客列表页 (/blog)
    └── [slug]/
        └── page.tsx # 博客文章页 (/blog/article-1)
```

## **3****.****1** **组件基础：构建UI的积木**

### **什么是组件？**



**组件是React和Next.js应用的基本构建块，它们是可重用的、独立的代码片段，用于构建用户界面。**

想象组件就像乐高积木：

​                ● 每个组件有特定的功能和外观（如按钮、卡片、表单）

​                ● 组件可以组合成更大的结构（如整个页面）

​                ● 组件可以接收不同的"参数"(props)，展现不同的样式和行为





 

 

### **组件的基本结构**

一个基本的React组件通常长这样：

 



```
// 简单的按钮组件
export default function Button({ text, onClick, color = "blue" }) {
  return (
    <button 
      onClick={onClick}
      className={`px-4 py-2 rounded text-white bg-${color}-500 hover:bg-${color}-600`}
    >
      {text}
    </button>
  );
}
```





 

使用这个组件：

 



```
// 在页面中使用按钮组件
import Button from '@/components/Button';
 
export default function HomePage() {
  return (
    <div>
      <h1>欢迎访问我的网站</h1>
      <Button text="点击我" onClick={() => alert('你好！')} color="green" />
    </div>
  );
}
```





###  

### **组件的优势**

组件带来的主要好处是：

​            1.     **可复用性**：我们可以在多个页面中使用相同的组件

​            2.     **可维护性**：修改组件会影响所有使用它的页面样式

​            3.     **关注点分离**：每个组件专注于特定功能

​            4.     **团队协作**：不同开发者可以处理不同组件

​            5.     **测试简化**：可以单独测试每个组件

## **3****.****2** **服务端组件与客户端组件**

Next.js 13引入了React的服务器组件，这是理解现代Next.js应用的关键概念。

###  

### **服务端组件 (默认)**

在App Router中，**所有组件默认都是服务端组件**，除非你明确指定它们是客户端组件。

 

**服务端组件的特点：**

​                ● 在服务器上渲染，将HTML发送给浏览器

​                ● 不包含任何JavaScript交互代码

​                ● 可以直接访问服务器资源（数据库、文件系统等）

​                ● 不能使用浏览器API、事件处理器或React hooks

 

**适用场景：**

​                ● 数据获取和处理

​                ● 访问服务器资源

// app/api/upload/route.js
import { writeFile } from 'fs/promises';
import { join } from 'path';
 
export async function POST(request) {
  const formData = await request.formData();
  const file = formData.get('file');
  
  if (!file) {
    return Response.json(
      { error: '没有提供文件' },
      { status: 400 }
    );
  }
  
  const bytes = await file.arrayBuffer();
  const buffer = Buffer.from(bytes);
  
  // 确保文件名唯一
  const fileName = `${Date.now()}-${file.name}`;
  const path = join(process.cwd(), 'public/uploads', fileName);
  
  try {
    await writeFile(path, buffer);
    return Response.json({ 
      success: true,
      fileName,
      url: `/uploads/${fileName}`
    });
  } catch (error) {
    console.error('上传错误', error);
    return Response.json(
      { error: '文件上传失败' },
      { status: 500 }
    );
  }
}

## **5****.****2** **部署前准备**

在部署到Vercel前，确保完成以下准备工作：

### **代码准备**

​            1.     创建远程仓库

​            2.     提交到远程仓库

 



```
# 确保所有更改都已提交到Git仓库
git add .
git commit -m "Ready for deployment"
git push
```





###  

### **环境变量准备**

准备生产环境的环境变量，本地的.env文件或者.env.local对应的是本地的环境变量。但如果你是生成环境，需要准备生成环境的环境变量，然后替换。

例如你本地的.env.local文件是：

 



```
# 示例
NEXT_PUBLIC_API_URL=http://localhost:3000 
```





 

需要准备一个用于生产环境的.env.production文件：



```
# 示例 
NEXT_PUBLIC_API_URL=http://your_domain.com 
```





 



生产环境是指用户真实的线上访问环境，开发环境是指你在本地开发的环境。你可以对应不同的环境变量。

## **5****.****3** **部署流程**

部署Next.js应用到Vercel非常简单：

###  

### **5****.****3.1 创建Vercel账户**

访问[Vercel](https://vercel.com/)，使用GitHub、GitLab或Bitbucket账户注册。

###  

### **5****.****3.2 导入Git仓库**

​            1.     在Vercel控制台点击"Add New… > Project"

​            2.     选择包含你Next.js项目的Git仓库

​            3.     Vercel会自动识别Next.js项目

###  

### **5****.****3.3 配置环境变量**

​            1.     在项目设置页面找到"Environment Variables"部分

​            2.     添加所有项目需要的环境变量，如Supabase连接信息（如果你的项目里面没有使用到环境变量，可忽略）

 



```
NEXT_PUBLIC_API_URL=https://hellonextjs-tutorial.vercel.app
 
# 其他环境变量
NEXT_PUBLIC_SUPABASE_URL=https://your-project.supabase.co
NEXT_PUBLIC_SUPABASE_ANON_KEY=your-anon-key
SUPABASE_SERVICE_ROLE_KEY=your-service-role-key
```





###  

### **5****.****3.4 部署项目**

点击"Deploy"按钮，Vercel将自动:

​                ● 克隆你的仓库

​                ● 安装依赖

​                ● 构建Next.js应用

​                ● 部署到全球网络

部署完成后，你会获得一个类似https://your-project.vercel.app的URL。

##  

## **5****.****4** **连接自定义域名**

### **添加域名**

​            1.     在项目控制台中，进入"Settings > Domains"

​            2.     点击"Add"并输入你的域名

​            3.     选择根域名或子域名

### **配置DNS**

Vercel提供两种方式配置域名:

​            1.     **使用Vercel DNS**：更改域名的名称服务器

​            2.     **使用外部DNS**：添加A记录和CNAME记录

配置完成后，Vercel会自动为你的域名颁发SSL证书，确保HTTPS访问。
